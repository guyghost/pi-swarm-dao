## 🐛 Problèmes de robustesse identifiés

### 1. Gates non activés
**Fichier:** `types.ts`  
**Problème:** `DEFAULT_CONFIG.requiredGates` ne contient que 4 gates :
```
["quorum-quality", "risk-threshold", "vote-consensus", "zone-compliance"]
```
**Conséquence:** Les gates suivants ne tournent jamais :
- `acceptance-criteria` ( Proposal #7 )
- `dependency-readiness` ( Proposal #14 )
- `dependency-conflict` ( Proposal #14 )
- `spec-completeness`, `delivery-feasibility`, `agent-registry-compliance`
- `self-amendment-safety`, `weight-conservation`, `prompt-integrity`, `circular-amendment`

**Recommandation:** Activer au minimum `acceptance-criteria`, `dependency-readiness`, `dependency-conflict`.

### 2. Dry-run et Execute s'écrasent mutuellement
**Fichier:** `delivery/dry-run.ts`  
**Problème:** `captureSnapshot()` utilise `state.snapshots[proposalId]` (clé unique). Quand `dao_execute` capture un snapshot, il écrase celui de `dao_dry_run`.
**Conséquence:** Impossible de rollback au pré-dry-run si l'exécution échoue.
**Recommandation:** Stocker les snapshots dans un tableau par proposal, ou ajouter un préfixe `dry-run-${proposalId}` vs `execution-${proposalId}`.

### 3. `dao_ship` référencé mais non implémenté
**Fichier:** `index.ts` (lignes 4054, 4184, 4201, 4534)  
**Problème:** Le help text et l'onboarding mentionnent `/dao:ship` comme commande disponible, mais aucun tool n'est enregistré.
**Conséquence:** UX cassée — les utilisateurs voient une commande qui n'existe pas.
**Recommandation:** Soit implémenter `dao_ship`, soit retirer les références.

### 4. Pas de synchronisation DAO state → GitHub labels
**Fichier:** `github-persistence.ts`  
**Problème:** Quand une proposition change de status dans le DAO state, les labels GitHub (`dao-status:open`, etc.) ne sont pas mis à jour automatiquement.
**Conséquence:** Les issues GitHub deviennent désynchronisées (ex: `#8` fermée mais label `dao-status:controlled`).
**Recommandation:** Appeler `ghUpdateStatus` après chaque transition de status.

### 5. `dao_dry_run` ne persiste pas `canProceed`
**Fichier:** `index.ts:4914`  
**Problème:** Le résultat `DryRunResult.canProceed` est retourné à l'utilisateur mais jamais stocké dans le state.
**Conséquence:** Le gate `mandatory-dry-run` (Proposal #15) ne peut pas vérifier si un dry-run a été fait.
**Recommandation:** Ajouter `dryRunAt?: string` et `dryRunCanProceed?: boolean` au type `Proposal`.

### 6. `runProposalExecution` ne vérifie pas le dry-run
**Fichier:** `index.ts:1702`  
**Problème:** La fonction `runProposalExecution` (appelée par `dao_execute`) ne vérifie pas si un dry-run a été effectué.
**Conséquence:** Proposal #15 (Mandatory Dry-Run Gate) est marquée comme exécutée mais le gate n'est pas implémenté.
**Recommandation:** Ajouter une vérification optionnelle avant `captureSnapshot`.
